## Decisions
- Wrap xterm.js container inside a native HTML outer and inner nested DivElement wrapper to separate padding margins from xterm.js column coordinate calculations.
- Bind click, keydown, and paste listeners dynamically during session attachment and ensure their subscriptions are cleanly disposed on session switch or unmount.
- Cleanly register and dispose xterm event subscriptions and ResizeObservers during the widget mount lifecycle to ensure no callbacks leak when unmounting or switching cached terminal sessions.
- Cap styled scrollback history loading to the last 200 lines to balance visual color richness and transport latency.
- Validate cache integrity upon container reuse, discarding broken or incomplete caches, and falling back to a fresh init.
- Re-attach custom key event handlers on every session attach to ensure key events are routed to the active widget/controller rather than capturing outdated states in cached terminal closures.
- Allow fitting on legitimately small or responsive split-pane layouts by lowering the minimum dimension check from 50px to 0px.
- Extract the visible/styled scrollback rows merge and history fetch logic into a unified shared helper in main.dart to eliminate code duplication and prevent path drift.
- Treat right-edge terminal spaces as padding unless `$` or `>` is the last visible prompt character; this prevents rows such as `> prompt        ` from placing the caret one cell too far right while still accounting for shell prompts that visually end in `$ ` or `> `.
- Use the transport-level `restore_session` request before attaching historical daemon sessions in the web client; attaching historical sessions directly leaves them read-only and makes the UI label them `exited`.
- Cached xterm.js instances need their input/resize subscriptions replaced atomically when a Flutter pane is remounted; otherwise session switching can leave a terminal with a stale or disposed handler.
- Resizing a restored daemon session must replay the raw PTY log into a fresh terminal at the requested size; resizing the existing terminal grid preserves old soft wraps and leaves restored history narrow.
- The web client must consume Snapshot/ResyncRequired events after resize; otherwise correct daemon snapshots are ignored and xterm continues rendering stale restored rows.
- Historical full-screen UI output cannot be safely reconstructed from styled row snapshots because cursor moves, clears, and alternate-screen transitions have already been flattened; live restored sessions should render the current viewport snapshot instead.
- Synthetic replay must not forward terminal-generated replies such as cursor position reports to the PTY; those bytes become visible shell input (`^[[1;1R`) and move the cursor out of place.
- Cursor placement tests need to cover the replay helper used by the web pane, not only local copies of the math in tests; otherwise the test suite can pass while xterm is left at the end of synthetic snapshot content.
- Real xterm replay can fail even when pure cursor math passes because long serialized rows soft-wrap in xterm and change the terminal row coordinates; browser-level tests should cover the actual xterm buffer cursor after replay.
- xterm can emit terminal reply data after the replay write callback; replay suppression must remain active briefly after the callback, or cursor-position replies can still reach the PTY as fragmented text.
- The replay cursor move is viewport-relative, so the web pane must write the same viewport slice used by the cursor calculation. Writing all fallback rows can scroll xterm before the cursor move and strand the cursor in blank history.
- Switching sessions remounts the web terminal pane; remote panes need to refresh their snapshot on selection before replaying, otherwise local backup rows accumulated during previous mounts can corrupt the restored view.
- Cache raw PTY log bytes up to 1MB in memory inside the daemon's OutputState to enable instant in-memory log reflow on resize without touching the disk.
- Debounce Flutter client PTY resize events by 100ms to throttle high-frequency layout changes and prevent I/O and process thrashing in the daemon.
- Pass includeHistory: true on session selection switch and daemon ResyncRequired events to ensure the client fully restores and reflows the complete historical scrollback buffer from the daemon.
- Replaying the active viewport slice alone in xterm.js leaves the scrollback buffer completely empty. Writing all historical rows first before the active viewport slice fills the scrollback buffer and keeps the viewport-relative cursor on the correct row.
- Snapshot events are broadcasted by the daemon on resize. Checking if the snapshot size is different from the last applied session size enables the client to conditionally fetch the reflowed history, preserving scrollback history on resize while avoiding slow round-trips for standard mock snapshots.
- Preserve and merge the active scrollback history rows from existing session rows when size-matching viewport-only snapshots are applied; this guarantees the client does not wipe out or overwrite its backup history buffer during live output updates.
- Queue WebSocket stream events in a sequential Future-based queue to ensure that asynchronous snapshot applications fully settle and update the sequence number before subsequent live output chunks are processed, eliminating race-induced double-prompt rendering issues.
- Bind a 150ms safety timer in the web terminal's synthetic replay suppression so that user input recovery is guaranteed even if xterm's async write callback fails to fire or gets delayed.
- Use a list-backed event queue processing loop to sequentialize asynchronous WebSocket events; this avoids complex Future chaining deadlocks and yields a resilient event handler that safely recovers from individual request errors.
- Automatically restore exited/historical sessions when they are selected or refreshed by the remote client to ensure they recover gracefully and become writable even if startup restoration failed.
- Only trigger terminal resets and synthetic replays when size changes, session selection refreshes, or resync is requested, completely eliminating cursor jump and prompt duplicate layout issues on size-matching snapshots.
- Append WebSocket output text to the last row of the client's backup log buffer if there is no preceding newline, instead of always adding a new row. This ensures the backup rows accurately match xterm.js row coordinates and prevents single-character fragments from corrupting replayed history.
- Capture window keydown events permissively unless standard inputs, textareas, or content-editable elements are focused, ensuring keyboard inputs route directly to the active terminal globally without focus-fighting.
- Avoid writing carriage returns (`\r`) before replaying the pending live buffer on fit triggers, maintaining the correctly computed cursor row/column alignment.
- Views with entirely empty/whitespace rows (as seen on newly restored sessions) should not clamp the PTY's actual cursor coordinates to 0,0, preventing caret coordinate misalignment and duplicate prompt flow issues.
- Manually route keyboard events when `shouldHandleTerminalKey` is true, ensuring terminal inputs are captured and forwarded to the active session globally even when focus shifts to the outer div containers inside Shadow DOM.
- Use standard RIS escape code (`\x1Bc`) and clear() instead of xterm.js `reset()` to securely reset terminal state, completely eliminating asynchronous type-safety crashes inside xterm's requestAnimationFrame routines.
- Simplify keyboard routing by manually capturing and forwarding window keydown events unconditionally whenever the terminal pane is active, completely eliminating fragile focus-state checks and native browser keydown blocking in Shadow DOM.
- Expose the active xterm.js instance on `window.activeTerm` for remote web client runtime diagnostics and end-to-end integration testing.
- Bypass websocket connection and pairing views in the web client using a `mock=true` URL query parameter to enable test suite deterministic execution.
- Compile and serve the web client in production release mode using a zero-dependency Node.js static file server for high-fidelity end-to-end browser integration tests, completely eliminating webdev/dwds injected debugging script crashes on Windows.
- Force xterm.js focused and inactive cursor styles to `'block'` to prevent the cursor from becoming faint or invisible in the browser when focus shifts to other parts of the application.
- Synchronously focus the active terminal on routed window keystrokes to reactivate cursor blinking immediately upon typing.
- Schedule delayed fit retries on session, widget, and click updates to ensure correct layout fitting when browser DOM dimensions are temporarily 0 during asynchronous platform view layout passes.
- Invoke native JS stringification for window selection objects on copy commands to avoid writing Dart stringified wrapper instances into the system clipboard.
- Use standard ANSI clear viewport and scrollback sequence (`\x1b[2J\x1b[3J\x1b[H`) for terminal screen clearing to prevent destructive RIS sequences from resetting private modes and option configurations.
- Defer xterm.js character fitting and initial scrollback/cursor replay until the stylesheet xterm.css has loaded (or a 600ms safety fallback triggers). This ensures that character cell measurements align perfectly with the browser styling, eliminating visual overlaps, incorrect themes, and staircasing prompts on the initial terminal load.
- Debounce the first initial log and cursor write by 150ms of stable width and height client dimensions. If dimensions change during mounting or flexbox layout transitions, the stability timer resets. This guarantees the first paint happens on a fully expanded, correctly sized canvas, avoiding narrow text wraps and reflow corruptions.
- Use `Queue<Map<String, dynamic>>` instead of `List` for client-side WebSocket event queuing to avoid slow O(n) shifts during `removeAt(0)` dequeues. This yields O(1) performance and keeps the event processing loop highly efficient under dense stdout output.
- Harden the static server traversal containment check by verifying resolved/normalized paths against `PUBLIC_DIR` boundaries, completely neutralizing sibling-directory traversal exploits.
- Wait 150ms after stylesheet onLoad before fitting. The browser requires a short rendering tick to parse the downloaded CSS and apply font metrics to the Shadow DOM; immediate fitting measures unstyled default fonts, leading to wrong column counts.
- Gate the initial log write to when the terminal is at least 35 columns wide. Extremely narrow intermediate layout sizes are skipped to prevent premature text wrapping.
- Gate PTY resize commands on `_initialContentWritten` during bootstrap. The browser DOM undergoes rapid reflow layout changes during initialization, triggering premature narrow fits (e.g. 53 columns). Gating and delaying the resize command ensures the daemon's ConPTY instance does not wrap the active prompt buffer into narrow columns before the client has expanded to its full stable size.
- Restore the terminal keyboard focus guard on global key handling. Window-level keydown routing should only handle events that originate from the terminal DOM path or while the pane focus node is active, preventing unrelated app text fields from forwarding input into xterm.
- Flush live output buffered during initial replay after the historical snapshot write. Clearing `_pendingLiveWriteBuffer` drops bytes that arrive during stylesheet/layout stabilization, so buffered chunks are now written to xterm immediately after initial content.
- Address current PR review comments by hardening the static server path resolution, removing per-keystroke logging, making WebSocket queue processing recover its guard in `finally`, and avoiding unconditional terminal focus activation from `build()`.
