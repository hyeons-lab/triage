## Decisions

- 2026-07-21T22:38-0700 Raise the toolchain pin rather than stay on cera 0.1.x. cera
  0.3.1 calls `vdotq_s32` throughout its NEON backend without declaring
  `feature(stdarch_neon_dotprod)`, having dropped that gate on the grounds that the
  intrinsic stabilized in 1.99.0-nightly. A feature gate only counts in the crate that
  declares it, so triage cannot re-add it from outside — the pin was the only lever.
  Confirmed with the user before changing it, since `rust-toolchain.toml` says to bump
  deliberately and the pin governs CI reproducibility for everyone.
- 2026-07-21T22:39-0700 Chose `nightly-2026-07-10` (1.99.0-nightly, af3d95584) over
  the newest nightly. Probing the channel manifests put the 1.98 → 1.99 boundary
  between 2026-07-05 and 2026-07-08, so this is close to the minimum that clears the
  floor. A three-month jump already risks new lints across the codebase; taking the
  smallest viable step keeps that risk down, and this toolchain was already installed
  locally so the verification run needed no download.
- 2026-07-21T22:44-0700 Propagate the new `new_session` error with `?` rather than
  unwrapping or falling back. Its docs derive the session's `ModalityCapabilities`
  from the manifest's `inference_type`, so a failure means the bundle cannot serve the
  requested modality — a real error. Both callers already return
  `anyhow::Result<Option<String>>` and already `?` on `append_text`/`generate`, so this
  matches the surrounding error handling exactly. `Ok(None)` would have been wrong:
  that value means "the model produced nothing usable", and is specified to preserve a
  good prior snippet rather than report a fault.

## Issues

- 2026-07-21T22:35-0700 The version bump alone produced 113 `E0658` errors inside
  cera's SIMD backend, which read as a broken release but are a toolchain-floor
  problem. Diffing the two crates' `lib.rs` made it obvious: 0.1.1 gates
  `stdarch_neon_dotprod`, 0.3.1 does not, and 0.3.1's own comment explains why. Worth
  remembering that an `E0658` from inside a dependency is usually a compiler-version
  mismatch, not a bad dependency.
- 2026-07-21T23:20-0700 Deploying to the live daemon ran the handover **twice**
  (06:15:19Z and 06:15:52Z), from one intended command. Both completed cleanly —
  `Old daemon committed to teardown (0x03)`, 13 sessions adopted each time, `gap_ms`
  23143 and 28486 against the 60s budget — so #113's slow-adoption fix held with cera
  0.3.1's model load in the path. The second handover began 3s after the first daemon
  finished loading its summarizer, which is a tighter back-to-back than the protocol
  normally sees. `--version` was ruled out as the trigger by reading `main.rs`: it parses
  to its own `Invocation::Version`, so #107 is intact.
- 2026-07-22T07:18-0700 **Resolved: the double handover was launchd, not the protocol.**
  The daemon runs as a `KeepAlive` LaunchAgent, so the manual `triaged --handover`
  killed launchd's own supervised job; launchd respawned it, and the respawned bare
  `triaged` hit #97's smart-start, found the manual process holding the socket, and
  handed over a second time. Three confirmations: the surviving daemon (PID 39294) has
  **PPID 1** and a start time of 06:15:52Z, i.e. the second handover's initiator *is*
  the launchd-spawned process; the gap from the old daemon's teardown commit
  (06:15:42.652Z) to handover #2 (06:15:52.829Z) is 10.177s, matching launchd's default
  10s `ThrottleInterval`; and `launchctl print` reports `runs = 18, pid = 39294`. This is
  the behaviour `main.rs:195-205` describes as intended — keying off "is one actually
  running?" is what stops a supervised respawn from fighting a manual deploy — so both
  handovers completing cleanly and the end state self-healing is the design working, not
  a fault. The cost is that the work is done twice (two adoptions, two model loads, ~62s
  of churn, two rolls of the timeout dice). The generalizable fact: **any manual
  `triaged --handover` on a machine with the service installed will always double-handover.**
- 2026-07-21T23:20-0700 Four PIDs that were children of the old daemon before the
  handover were gone afterwards, and four fresh shells appeared at 06:16:28–43Z with
  restored cwds. Whether any live session shell was actually lost is **unresolved**: the
  pre-handover snapshot recorded PIDs only, not command names, and the daemon spawns
  short-lived `git` children for repo context (#83), so the four could equally have been
  transient helpers exiting normally. Eight shells provably survived with 6.5h uptime.
  `demote_dead_live_session` (`session.rs:887`) exists precisely for "a session that dies
  while live — notably one adopted across a handover", demoting it to `Historical` so it
  can be re-spawned; respawn-with-restored-cwd is the designed recovery, not a crash. To
  settle it, capture `pid,command` (not just `pid`) before a single handover and diff.
- 2026-07-22T07:32-0700 **Resolved: no shell was lost, and the original measurement was
  invalid.** Instrumented adoption on `fix/handover-session-logging` and re-ran the
  handover. Session shells are **not children of the adopting daemon** — the process that
  forked them exits during teardown, so they reparent to PID 1; only sessions spawned
  *after* the last handover are the daemon's own children. Counting
  `ps --ppid <daemon>` therefore undercounts adopted sessions by construction, which is
  the entire reason the 2026-07-21 snapshot looked like four vanished shells. In the
  controlled run all 13 sessions were adopted twice with identical PIDs, and 10 of the 13
  shells are alive with PPID 1 and ~22h uptime, having now survived several handovers.
  The three exceptions (`session-73/80/104`) were logged as `session child exited` within
  1–3ms of adoption in **both** handovers — already dead beforehand, not killed by it.
- 2026-07-22T07:32-0700 That result exposed a genuine defect worth its own change: because
  demotion is lazy (only on a restore request), the daemon keeps sessions whose shells died
  long ago in the `Live` map indefinitely, serializes them into every handover, and passes
  PTY fds for dead processes. The on-disk manifest compounds it — all 32 sessions are
  persisted `exited: false`, covering both the 19 that are merely `Historical` and the 3
  `Live` ones with confirmed-dead shells, so at least 22 are recorded live without a
  running shell. "Adopting 13 inherited live sessions" overstates what is live for the
  same reason, and the count has been the only signal available.
- 2026-07-22T07:35-0700 Recorded the parent commit in **Commits** as a hash that does not
  exist. The value was copied out of tool output where the real SHA had been replaced by a
  redaction placeholder, so it looked like a plausible short hash and was transcribed as
  one; `git cat-file -t` returned "Not a valid object name". Corrected to `023cf1d`. Any
  hash, path, or identifier that will be written down has to come from a command run for
  that purpose and be verified against the object store, not lifted from surrounding
  output.
- 2026-07-22T07:18-0700 Nearly used "no demotion lines in `triaged.log`" as evidence that
  no session shell was lost across the handover. It is not evidence:
  `demote_dead_live_session` logs nothing on its success path — the only `tracing` call in
  it is a `warn!` for a failed actor reap — and it is invoked lazily from exactly one site
  (`session.rs:1672`, the restore request), so a session can be demoted without the log
  ever mentioning it. Silence from a path that does not log is not a negative result. The
  shell question therefore stays open, and the controlled `pid,command` run is still what
  settles it; the missing observability is itself worth fixing.

## Research & Discoveries

- The 1.98 → 1.99 nightly boundary falls between 2026-07-05 (1.98.0-nightly, c397dae80)
  and 2026-07-08 (1.99.0-nightly, f10db292a), per the `channel-rust-nightly.toml`
  manifests under `static.rust-lang.org/dist/<date>/`. Useful for choosing a minimum
  pin instead of guessing at a date.
- cera 0.3.1's feature table is compatible with 0.1.1 for triage's purposes: `remote`
  expands identically, and the default set is unchanged. Only `gpu` differs (adds
  `dep:futures-channel`), which triage does not enable.
- The toolchain pin lives in five places: `rust-toolchain.toml` plus four hardcoded
  `toolchain:` keys across `ci.yml` and `publish.yml`. Only the first is authoritative
  for local builds, so a partial bump fails exclusively in CI.
- `cargo check` surfaces `proc-macro-error2 v2.0.1` as containing code a future Rust
  will reject. It is transitive, not triage's code, and does not fail any gate — but it
  will eventually force either an upstream fix or dropping whatever pulls it in.
- The `cached file size mismatch; re-downloading` line the daemon logs on every start
  (`Q4_0.json`, `expected 296`, `actual 1878`) is **not** from this bump — it appears
  with identical numbers back to 2026-06-21 under cera 0.1.1. cera's `bundle/download.rs`
  is byte-identical between the two versions. It re-fetches a 1.8 KB manifest, not the
  model, so it costs a round-trip rather than a re-download, but the expected size looks
  permanently stale in whatever records it.
- Two deviations from the plan, both additions rather than changes of approach: the
  publish-job action pins and the README nightly note. Neither was foreseen when the plan
  was written; both came out of the review pass.

## Lessons Learned

- A dependency that drops a `#![feature(...)]` gate silently raises its consumers'
  toolchain floor without changing its declared `rust-version`. cera 0.3.1 still says
  `rust-version = "1.85"`, which is unachievable on aarch64 — the manifest cannot
  express "needs 1.99 on this target", so the real floor is only discoverable by
  compiling.
