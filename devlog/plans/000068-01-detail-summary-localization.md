## Thinking

The side-rail hover popover shows a longer-form "detail" summary generated by the
local LLM (`crates/triaged/src/summarizer.rs`, `generate_detail`). Today it is a
purely activity-focused 2–3 sentence blurb produced from the session's current
screen. It has no localization metadata, so when many sessions look similar
(several `cargo test` runs, several agents) the summary alone doesn't tell the
user *which* session it is.

The daemon already knows the authoritative git context per session
(`SessionContext { repository_root, worktree_root, branch }`, resolved in
`resolve_session_context`) and the client already renders a `repo · branch ·
worktree` meta line in the rail tile. We want the detail summary itself to carry
that same localization so it is self-contained (and so the future session-search
corpus indexes it).

Key design choice: the repo/branch/worktree must be **deterministic and exact**.
A 1.2B local model would happily hallucinate a branch name. So we do NOT ask the
model for them — we prepend a deterministic header built from `SessionContext`,
mirroring the client's meta-line convention exactly (omit absent parts; hide the
worktree leaf when it equals the repo root or merely echoes the branch). The
model is told explicitly not to guess git fields and to focus on activity, and
its length budget is relaxed (up to ~5 sentences) so it can describe enough to
localize (command, files, error text, state).

Second ask: honour the LeapBundles manifest's advisory `sampling_parameters`
when the loaded model is a leap text bundle. cera exposes these via
`engine.manifest().generation_defaults` (`GenerationDefaults::Text`). We apply
temperature/top-p/top-k/repetition-penalty from the manifest for the *detail*
pass, falling back to greedy (temp 0) when absent. The one-line label stays
greedy regardless, so the terse rail label is stable across regenerations
(per user request: "apply temp-0 to one liner so it's consistent").

Plumbing: `SummarizeJob` gains `context: Option<SessionContext>`. The two
enqueue sites already call `summary_rows()`, which does an off-lock actor
round-trip; extend the `SummaryRows` actor response to also carry
`self.context.clone()` so no extra round-trip is needed.

## Plan

1. `summarizer.rs`
   - Add `context: Option<SessionContext>` to `SummarizeJob`.
   - Rewrite `DETAIL_SYSTEM_PROMPT`: localization-focused, up to ~5 sentences,
     explicit "do not guess git repo/branch/dir".
   - `generate_detail(engine, config, prompt_text, context)`: prepend
     `context_header(ctx)` to the sanitized model output; return header-only
     when the model is empty but context exists.
   - `sampling_opts(engine, max_tokens)`: manifest-driven sampling for the
     detail pass; greedy fallback. One-line pass stays explicit temp-0.
   - `context_header` + `leaf_name` helpers mirroring the client meta line.
   - Raise `MAX_DETAIL_CHARS` 280 → 480 (caps only the model portion).
   - Unit test `context_header_mirrors_the_side_rail_meta_line`; extend the
     ignored e2e test to assert the header leads the detail.

2. `session.rs`
   - Extend `ActorCommand::SummaryRows` response to
     `(Vec<String>, u64, Option<SessionContext>)`; handler returns the context.
   - `summary_rows()` / `request_summary_rows()` return types updated.
   - Both enqueue sites populate `SummarizeJob.context`.

3. `config.rs`
   - `detail_max_tokens` 110 → 180 to fit the longer summary.

4. Verify `cargo check` + `cargo test -p triaged summarizer`.
