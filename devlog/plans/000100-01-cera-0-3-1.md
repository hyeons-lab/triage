# Plan — bump cera to 0.3.1

## Thinking

The ask is a dependency bump: `cera` 0.1.1 → 0.3.1, freshly published. Two 0.x
minor boundaries, so breaking changes are expected by semver convention.

**Feature surface is unchanged.** The workspace pins
`cera = { version = "0.1.1", features = ["remote"] }`. Comparing the feature
tables from crates.io, `remote` still exists in 0.3.1 with the same
`dep:reqwest`/`dep:sha2`/`std-fs` expansion, and the default set is identical.
Only `gpu` differs (0.3.1 adds `dep:futures-channel`), which triage does not
enable. So no feature-flag work.

**Usage is confined to one file.** `crates/triaged/src/summarizer.rs` is the only
source file mentioning cera. It touches `CeraEngine`, `EngineConfig`,
`BackendPreference`, `SessionConfig`, `GenerateOpts`, `manifest::GenerationDefaults`,
`session::{FinishReason, ModalitySink, CeraError}`,
`tokenizer::{BpeTokenizer, ChatMessage, apply_chat_template}`, and
`bundle::BundleRepo`. A narrow surface, so the compiler can enumerate the breakage
rather than needing an upfront changelog audit.

**The blocking problem is the toolchain, not the API.** With only the version
bumped, cera fails to compile with 113 `E0658` errors: `stdarch_neon_dotprod` is
an unstable library feature and cera no longer enables it. cera 0.1.1's `lib.rs`
declared
`#![cfg_attr(target_arch = "aarch64", feature(stdarch_neon_dotprod, stdarch_aarch64_prefetch, stdarch_neon_i8mm))]`;
0.3.1 drops `stdarch_neon_dotprod` from that list, with a comment stating it
stabilized in 1.99.0-nightly and that keeping the gate would trip the
`stable_features` lint. But 0.3.1 still calls `vdotq_s32` throughout that backend. So on any
nightly older than 1.99, the calls are unguarded uses of an unstable intrinsic.

triage pins `nightly-2026-04-02` = 1.96.0-nightly, three releases below the floor.
This is not something triage can patch around: the feature gate has to be declared
in the crate that uses the intrinsic, and that is cera. The only options are to
raise the pin or to stay on cera 0.2.x/0.1.x.

Probing the nightly channel manifests puts the 1.98 → 1.99 boundary between
2026-07-05 and 2026-07-08. `nightly-2026-07-10` (1.99.0-nightly, af3d95584) is the
chosen pin — the smallest jump that clears the floor, which also minimizes new-lint
churn from a three-month toolchain jump. Nightly is still required regardless:
`stdarch_neon_i8mm` and `stdarch_aarch64_prefetch` remain gated in 0.3.1.

Note that cera 0.3.1 declares `rust-version = "1.85"`, which is not achievable on
aarch64 for the reason above. Worth reporting upstream, but not triage's problem to
fix here.

**The real API break is one signature.** `CeraEngine::new_session` went from
returning `Session` to `Result<Session, CeraError>` — its docs say the session's
`ModalityCapabilities` is derived from the manifest's `inference_type`, so
construction can now fail. Four errors from two call sites (`generate_one_line`,
`generate_detail`). Both callers already return `anyhow::Result<Option<String>>`
and already `?` on other cera calls, so propagating is both correct and consistent:
an engine that cannot open a session is a genuine failure, not something to paper
over with a default.

## Plan

1. Bump `cera` to `0.3.1` in `[workspace.dependencies]`, keeping `features = ["remote"]`;
   refresh `Cargo.lock` with `cargo update -p cera`.
2. Raise the toolchain pin to `nightly-2026-07-10` in `rust-toolchain.toml`, and
   rewrite its header comment: `stdarch_neon_dotprod` is no longer a reason for
   nightly, the remaining reasons are i8mm/prefetch, and the 1.99 floor plus its
   cause should be recorded so the pin is not lowered casually.
3. Update the same pin in the four places CI hardcodes it — `.github/workflows/ci.yml`
   (×2) and `.github/workflows/publish.yml` (×2). These install by name rather than
   reading `rust-toolchain.toml`, so a missed one fails CI.
4. Add `?` to both `engine.new_session(...)` call sites in `summarizer.rs`.
5. Verify with the full CI gate set, not just tests: `cargo fmt --all --check`,
   `cargo clippy --workspace --all-targets --all-features -- -D warnings`,
   `RUSTDOCFLAGS="-D warnings" cargo doc --workspace --all-features --no-deps --locked`,
   and `cargo test --workspace`. The toolchain jump is the risk here — a three-month
   newer nightly can surface lints across the whole codebase, independent of cera.
6. Confirm no doc or config outside the devlog still names the old toolchain or the
   old cera version.
